# TaskForge 事故案例库

> 每条规则背后的真实事故。遇到类似场景时应主动查阅，避免重蹈覆辙。

## 规则 1：批量操作必须先试跑再全量

### 案例 1.1：AST 迁移脚本 v3 损坏 282 文件

**时间**: 2026-07-05
**影响**: 2,619 处位置损坏，11,011 个 TS 编译错误

脚本对 `t()` 调用做 AST 级批量替换。虽然做了 3 文件试跑且通过，但这 3 个文件恰好没有"紧密相邻的多 StringLiteral 替换"场景。ts-morph 的 `getStart()/getEnd()` 返回未修改源文件中的位置，批量替换导致位置偏移——后续替换使用了过期的位置信息。

**根因**: 试跑样本未覆盖边缘情况就默认全量安全。"先全量跑再说"的心态。

**正确做法**: 试跑样本必须包含紧密相邻的多替换场景；替换前先写临时文件；替换后立即 `tsc --noEmit` 验证。

### 案例 1.2：MEDIUM 安全修复导致 8 文件功能崩溃

**时间**: 2026-07-14
**影响**: frontend 服务丢失端口配置、monitoring 服务端口误入 networks 列表、SSE 崩溃

一次性修改 docker-compose.yml、api.ts 等 8 个文件，未做功能性回归测试。修改 `docker-compose.yml` 时用片段拼接而非完整 service 替换，导致前端服务端口配置丢失。修改 `api.ts` 时使用不存在的 `ky.extendRequest()` API，SSE 功能直接崩溃。全部回滚。

**根因**: 批量修改无回归验证，不验证 API 存在性就使用。

**正确做法**: docker-compose.yml 用完整 service 替换；修改后至少做 `docker-compose config` + TS 编译验证。

### 案例 1.3：Apple HIG 正则替换 JSX 全部返工

**时间**: 2026-07-05
**影响**: 200+ 文件正则替换结果全部废弃

用 Python/Shell 正则做 JSX 代码中的 `var(--*)` 引用替换，完全无法识别 JSX 上下文语义。正则匹配了注释、字符串、对象属性等不该替换的位置。

**根因**: 正则无法理解代码语义，JSX 替换必须用 AST 工具。

**正确做法**: 禁止用正则替换代码文件，必须用 ts-morph AST transformer。

## 规则 2：CI 门禁是必须基础设施

### 案例 2.1：818 个 TS 编译错误合入 main

**时间**: 2026-07-05
**影响**: Apple HIG 迁移后，818 个 TS 编译错误的代码被提交到 main 分支

没有 CI 的 `tsc --noEmit` 检查阻止错误代码合入。开发者本地未做编译检查就 push。

**根因**: 无自动化质量门禁，一切依赖人工 review。人工 review 在大代码库（2247 API 端点、145+ 页面）面前完全失效。

### 案例 2.2：安全漏洞长期潜伏

**时间**: 2026-07-14 发现
**影响**: 5 CRITICAL + 11 HIGH 安全漏洞长期存在

.env 含明文 Fernet 主密钥、docker-compose 硬编码极弱密码、端口 0.0.0.0 暴露加默认密码。无安全扫描的情况下这些问题长期存在。

**根因**: 没有安全扫描集成，安全漏洞只能靠人工发现。

### 案例 2.3：2% 测试覆盖率无人关注

**影响**: 核心财务/Agent/LLM 路径几乎无测试覆盖

覆盖率从 2% 起步，没有门禁阈值，差距无法被感知。

**根因**: 无覆盖率门禁，低覆盖率没有开发痛感。

---

## 规则 3：前后端必须有共享接口契约

### 案例 3.1：E-commerce 13 个端点全部 404

**影响**: 前端 `ecommerce.ts` 调用 13 个专用端点，后端仅有 6 实体 CRUD（30 个通用端点），专用业务端点全部不存在

用户访问电商场景页直接崩溃。前后端无共享接口定义，开发者各自为战。

**根因**: 无 OpenAPI schema 自动生成 TypeScript 类型，无契约测试。

### 案例 3.2：427 个孤立后端端点

**影响**: 427 个后端端点无前端调用，大量后端能力"建了但没人用"

**根因**: 新增后端端点无对应前端调用的强制要求，也无"API-only"标记机制。

### 案例 3.3：8 个场景前端缺 U/D 操作 70-80%

**影响**: Deals 0% 缺失，Caster/Handmade/Coach/Consultant 缺失 80%

**根因**: 前后端独立开发，无 CRUD 覆盖率检查。

---

## 规则 4：模块完成度必须可验证

### 案例 4.1：2 个引擎目录完全缺失

**影响**: `src/engine/caster/` 和 `src/engine/freelance/` 目录不存在，health 端点硬编码 `engine_ok: True`

**根因**: 骨架代码写了目录和文件但没完成，health 基于硬编码而非实际检查。

### 案例 4.2：4 个引擎 __init__.py 导出断裂

**影响**: `office/__init__.py` 和 `handmade/__init__.py` 是空文件（ImportError），`finance/__init__.py` 导出旧模块类

**根因**: 零导出的 `__init__.py` 没有被检测机制发现。

### 案例 4.3：CROSS_ENGINE_FK_REGISTRY 仅覆盖 4/11 场景

**影响**: 缺 7 个场景的外键注册，跨引擎查询可能返回不完整数据

**根因**: FK 注册无完整性检查机制。

---

## 规则 5：安全默认值必须安全

### 案例 5.1：.env 明文 Fernet 主密钥

**影响**: 密钥与加密数据同文件，泄露可导致全量数据解密

### 案例 5.2：docker-compose 硬编码极弱密码

**影响**: calcom:calcom, ninja:ninja 等弱密码直接写在配置文件中

### 案例 5.3：Chat 缓存硬编码默认口令

**影响**: `taskforge-default-passphrase-v1` 作为加密口令硬编码在代码中

### 案例 5.4：localStorage 明文存储 AES-GCM 加密口令

**影响**: 加密口令以明文存储在浏览器 localStorage 中

### 案例 5.5：PaddleOCR 容器以 root 运行

**影响**: 唯一未设 user 的服务，容器逃逸风险

**共同根因**: 开发环境"先跑起来"的便利性压倒了安全要求。没有安全基线检查，没有 .env.example 与 .env 的强制区分。

---

## 规则 6：测试是开发的一部分

### 案例 6.1：2% 测试覆盖率

**影响**: 核心路径几乎无覆盖，重构极度困难，安全修复无法验证回归

### 案例 6.2：测试隔离问题

**影响**: `test_tool_result_persist.py` 单跑 33 passed，全量跑被前序测试污染失败（环境变量/单例残留）

### 案例 6.3：测试先行但实现缺失

**影响**: `test_v2_p1_p2.py` 测试了 ContextEngine 抽象，但代码未落地——只有测试没有实现

### 案例 6.4：前端零测试框架

**影响**: devDependencies 为空，TypeScript 错误只能在运行时发现

**共同根因**: 测试不是开发流程的一部分，是"事后补"的。

---

## 规则 7：批量修改必须原子性

### 案例 7.1：AST 迁移脚本无回滚

**影响**: ts-morph `saveSync()` 在某些情况下保存空内容，文件直接丢失

**根因**: 无"先写临时文件再替换"的安全机制。

### 案例 7.2：docker-compose.yml 片段拼接

**影响**: frontend 服务丢失端口配置、monitoring 服务端口误入 networks 列表

**根因**: 片段拼接而非完整 service 替换，中间状态不完整。

### 案例 7.3：ky.extendRequest() API 不存在

**影响**: SSE 功能直接崩溃

**根因**: 不验证 API 存在性就使用。

---

## 规则 8：新技术引入前必须 POC 验证

### 案例 8.1：正则替换 JSX 全部返工

**影响**: 200+ 文件替换结果全部废弃

**根因**: 未做 POC 验证正则能否正确处理 JSX 语义。

### 案例 8.2：AST 位置偏移 bug

**影响**: 紧密相邻的多 StringLiteral 替换位置偏移

**根因**: POC 试跑样本恰好没有覆盖这种边缘情况。

### 案例 8.3：ky.extendRequest() 不存在

**影响**: SSE 崩溃

**根因**: 使用 API 前未读官方文档确认存在性。

---

## 规则 9：敏感数据必须与代码库隔离

### 案例 9.1：3.4GB 数据库差点打包

**影响**: `data/taskforge.db` 3.1GB 含用户表/refresh_tokens/审计日志

### 案例 9.2：飞书凭证明文存储

**影响**: `data/secrets/builtin/feishu.json` 含明文 app_secret + open_id

### 案例 9.3：根目录 50+ 临时文件

**影响**: 调试脚本、测试截图、临时备份、交付物散落在根目录

### 案例 9.4：.env 含明文密钥

**影响**: Fernet 主密钥与加密数据同文件

**共同根因**: 开发数据、测试数据、敏感凭证没有与代码库隔离。

---

## 规则 10：代码模式必须强制一致

### 案例 10.1：3 种 health 端点格式

**影响**: 
- A) 标准 `{scenario, tables_ok, engine_ok}`（9 个场景）
- B) 硬编码 `{scenario, tables_ok:True, engine_ok:True}`（3 个）
- C) 旧格式 `{scenario, status:"ok"}`（2 个）

前端无法统一处理。

### 案例 10.2：stats() 签名不一致

**影响**: 有 `days=30` 的、无参数的、用 `platform=None` 的

### 案例 10.3：Booking 重复 service 层

**影响**: `scenario-booking.ts`（7 函数）与 `pages/Booking/services.ts`（9 函数）功能重叠，签名略异

### 案例 10.4：4 个场景组件"无家可归"

**影响**: Coach/Consultant/Handmade/Caster 有视觉组件但无 Tab 页面承载

**共同根因**: 没有代码规范强制执行机制，开发者按各自理解实现。
